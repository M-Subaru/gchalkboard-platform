What this fixes and why:
The old matcher /((?!_next/static|...).*)  was matching every URL — home page, about, for-teachers, post-vacancy, every API call, everything. Each hit triggered supabase.auth.getUser(), which is an outbound HTTPS request to Supabase. On Cloudflare Workers free tier the CPU clock runs during that network wait, exhausting the 10ms budget almost immediately.
The new matcher only fires on /dashboard/*, /teachers/*, /schools/*, and /login — the only routes that actually need auth state. Public pages (/, /about, /for-teachers, etc.) are served directly from the ASSETS binding with zero CPU overhead. API routes handle their own auth internally.
After this deploy the 1102 errors should stop entirely. The Cloudflare CPU chart should drop from ~60ms back down to low single digits for typical visits.
